# 06. Экран Debug Commands

## Назначение
Экран `DebugCommandsActivity` позволяет вручную отправлять зафиксированные в логах команды на подключённые очки после перехода в состояние READY/BONDED. Он повторно использует существующий `BleManager`/`BleLogger` через `BleSessionHolder` и не трогает низкоуровневый Bluetooth API напрямую.

## Предустановленные команды
Список формируется в `DebugCommandRepository` и совпадает с полезными нагрузками из `1_phase_pair_connection_log.json`:
* **Start handshake** — `00 00 06 11 01 00` (стартовый write на handle 0x0023).
* **Auth challenge 1/2** — два крупных блока `00 00 02 10 0a 06 ...` отправленных на тот же handle.
* **Settings block** — длинный JSON-пакет `00 00 08 00 01 00 02 1a 0c 65 ...` (handle 0x0026).
* **Settings commit** — короткое подтверждение `01 00 37 63 30 ...` (handle 0x0026).
Любую команду можно отредактировать перед отправкой; флаг With response переключает write type.

## Как добавить новые
1. Найти байтовую последовательность в свежих логах.
2. Добавить новый `Command` в `DebugCommandRepository.commands` с понятным именем и HEX-полезной нагрузкой.
3. Пересобрать приложение — элемент появится в списке.

## Использование для реверса
* Отправить стартовый пакет и убедиться, что уведомления (RX) появляются в нижнем лог-вью — они приходят через общий `BleLogger`.
* Менять отдельные байты в HEX-поле и наблюдать, меняется ли ответ/поведение устройства.
* Сравнивать полученный RFCOMM-трафик в классическом канале с теми же JSON-блоками из BLE, чтобы восстановить соответствие команд.
